如果有些文件已经被忽略了，当使用git add时是无法添加的。需要加上-f参数才能强制添加到git中去，这样就能强制添加到缓存中去了。

如果我们意外的将想要忽略的文件添加到缓存中去了，我们可以使用rm命令将其从中移除：

在 .gitingore 文件中，每一行指定一个忽略规则，Git检查忽略规则的时候有多个来源，它的优先级如下（由高到低）：

如果你不慎在创建.gitignore文件之前就push了项目，那么即使你在.gitignore文件中写入新的过滤规则，这些规则也不会起作用，Git仍然会对所有文件进行版本管理。简单来说出现这种问题的原因就是Git已经开始管理这些文件了，所以你无法再通过过滤规则过滤它们。所以大家一定要养成在项目开始就创建.gitignore文件的习惯。

gitignor忽略规则查看 如果你发下.gitignore写得有问题，需要找出来到底哪个规则写错了，可以用git check-ignore命令检查

聊聊后端：

Django:

URL配置：

  • 根 URL 配置：用于项目范围的 URL 映射，包括项目的主页、管理员界面、通用视图等。通常情况下，根 URL 配置文件只有一个，处理项目的顶级 URL 映射。通常位于项目的根目录下，可在项目的设置文件（settings.py）中通过 ROOT_URLCONF 设置指定
  • 应用级别 URL 配置：用于应用内的 URL 映射，包括应用的特定功能和视图。每个应用都可以有自己的 urls.py 文件，以定义应用内的 URL 路由。这有助于模块化应用，并将相关的功能组织在一起。

include函数用于包含其他应用的 URL 配置，可以将其他应用中定义的 URL 路由包含到当前应用的路由中，实现模块化的 URL 配置。相当于套娃。

as_view()

  1. 在导入类视图时，通常是类似 UserRegisterApi 这样的视图类。
  2. 使用 .as_view() 方法将类视图转换为可调用的函数。
  3. 在 URL 配置中，将这些可调用的函数与特定的 URL 路由路径关联，这样当请求到达该 URL 时，Django 将调用相应的函数处理请求。

变模型需要这三步：

  • 编辑 models.py 文件，改变模型。
  • 运行 python manage.py makemigrations 为模型的改变生成迁移文件。
  • 运行 python manage.py migrate 来应用数据库迁移。这个 migrate 命令选中所有还没有执行过的迁移（Django 通过在数据库中创建一个特殊的表 django_migrations 来跟踪执行过哪些迁移）并应用在数据库上 - 也就是将你对模型的更改同步到数据库结构上

数据库迁移被分解成生成和应用两个命令是为了让你能够在代码控制系统上提交迁移数据并使其能在多个应用里使用；这不仅仅会让开发更加简单，也给别的开发者和生产环境中的使用带来方便。

每个视图必须要做的只有两件事：返回一个包含被请求页面内容的 HttpResponse 对象，或者抛出一个异常，比如 Http404 。至于你还想干些什么，随便你。

你的视图可以从数据库里读取记录，可以使用一个模板引擎（比如 Django 自带的，或者其他第三方的），可以生成一个 PDF 文件，可以输出一个 XML，创建一个 ZIP 文件，你可以做任何你想做的事，使用任何你想用的 Python 库。

序列化：

  • 序列化是将模型数据转换为可在网络上传输或存储的格式的过程。这通常意味着将模型实例转换为JSON、XML或其他格式，以便客户端能够读取或显示数据。
  • 序列化通常在读取（查询数据库并将数据发送到客户端）时使用。它将数据库中的模型数据转换为可传递给客户端的数据。

反序列化：

  • 反序列化是将接收到的数据（例如，客户端提交的JSON数据）转换回模型数据的过程。这包括验证数据，并确保数据符合模型的规则和字段要求。
  • 反序列化通常在创建、更新或验证数据时使用。它将来自客户端的数据转换为可以用于创建或更新数据库中的模型实例的数据。

10-08 CSDN太牛逼了挖草，我卡了一个国庆的数据库迁移版本历史bug，给他两句话解决了…… 具体什么个情况呢？我makemigrations生成需要进行迁移的任何新的或已更改的模型信息没有问题，但是当我应用的时候就出现了报错：raise InconsistentMigrationHistory( django.db.migrations.exceptions.InconsistentMigrationHistory: Migration admin.0001_initial is applied before its dependency account.0001_initial on database ‘default’.

原因：Django中有一个原生的User模型类，admin的模型依赖这个模型类，由于前面一个应用中的模型类User继承了AbstractUser类，所以提示这个错误。

解决方案：利用sqlite3的交互面板dbshell，利用manage.py,先导出.table，然后全部drop掉
