python新语法

上述代码的实际效果是一样的，第一行只是使用了python的类型注释。使用冒号是一种类型注解的语法。在Python 3.6及以后的版本中，可以使用类型注解来指定变量的类型。在这里，正在将 html 这个变量标注为 Selector 类型。这对于代码的可读性和类型检查非常有帮助，但不是必需的。

Xpath

bodys = html.xpath(‘//a[@target=”_blank”]/@href’).extract()

这行代码使用 XPath 表达式来定位 HTML 中的链接 (<a> 元素的 href 属性)。下面解释一下这个 XPath 表达式的各个部分：

  1. //a: 选择所有的 <a> 元素，不论其在 HTML 文档中的位置。
  2. [@target="_blank"]: 选择具有 target 属性值为 “_blank” 的 <a> 元素。
  3. /@href: 选择选定的 <a> 元素的 href 属性。

综合起来，这个 XPath 表达式会选择所有具有 target 属性值为 “_blank” 的 <a> 元素，并返回它们的 href 属性的值，即链接的 URL。

/following-sibling::text()[1] 是XPath中的一个表达式，用于在当前节点的后继兄弟节点中定位文本节点

解释如下

  1. def get_body(self, response: Response):

    • 这是一个自定义的方法定义，它接收一个名为 response 的参数，该参数是一个 Scrapy 的 Response 对象，表示从网页获取到的响应。

  2. html_dict = json.loads(str(response.body, encoding="utf-8"))

    • 这一行将响应的内容（字节流）转换为字符串，然后使用 json.loads() 方法将其解析为 Python 字典。这里假设响应内容是 JSON 格式，其中包含一个名为 articleListHtml 的键，其对应的值是 HTML 文本。

  3. html_text = html_dict.get("articleListHtml", None)

    • 从上一步得到的字典中，使用 get 方法获取键为 "articleListHtml" 的值，即包含 HTML 文本的部分。如果键不存在，则返回 None。

  4. html: Selector = Selector(text=html_text)

    • 这一行创建了一个 Scrapy Selector 对象，用于解析 HTML 文本。Selector 是 Scrapy 提供的用于处理网页内容的工具，可以使用不同的选择器方法（如 XPath 或 CSS 选择器）来提取所需的数据。

    从 JSON 响应中提取出包含 HTML 内容的部分，然后使用 Selector 对象对该 HTML 文本进行解析

xpath的新用法：

首先还是得明确response.xpath 返回一个 SelectorList 对象

  1. rows = response.xpath("//tr"): 这行代码使用 XPath 表达式 //tr 定位到网页中所有的 <tr> 元素，即表格的每一行。rows 是一个 SelectorList 对象，其中包含匹配的所有行元素。
  2. columns = row.xpath(".//td"): 在每一行中，通过 XPath 表达式 .//td 定位到该行内所有的 <td> 元素，即表格的每一列。columns 是一个 SelectorList 对象，其中包含匹配的所有列元素。
  3. row_data = [column.xpath("string(.)").get().strip() for column in columns]: 在每一列中，通过 XPath 表达式 string(.) 提取该列内的文本内容。.get() 方法获取文本值，strip() 方法去除文本中的空白字符。这行代码创建了一个列表 row_data，其中包含了该行内每个单元格的文本内容。
  4. target_date = datetime.strptime(row_data[-1], '%Y-%m-%d').date(): 这行代码将该行中最后一列的日期字符串（格式为 YYYY-MM-DD）转换为 datetime.date 对象，以便后续.

完成更改前：

我的流程：

1.Fork 项目仓库：去上游老板那里，需要 Fork 项目的原始仓库到你自己的 GitHub 账号下。这将创建一个你自己的仓库副本。

2.克隆仓库： 将你 Fork 后的仓库克隆到本地。

3.创建分支： 在本地仓库中创建一个新分支，用于你的修改。 这一步看你的项目规模已经合作方式，可有可无

4.进行修改： 添加新功能、修复 bug 等。

当你完成了更改。我先陈述一下我的流程：

1.首先是git fetch upstream upstream是我们小组建立的远程仓库别名，我先抓取文件更新下载到本地。可以使用git remote 查看仓库，git remote show upstream 来查看具体信息，如果你忘了的话

2.可以直接merge，但是我怕会把我未提交的merge掉，所以我求稳，我在merge前我会先使用git status查看状态

3.比如说我当前的状态是多处修改没有提交，那么我就会选择部分文件为一组提交。使用git add 部分文件的路径（这时候，git status的优势就体现出来了，因为你可以直接cv）使用git add后如果不小心加错了可以使用git reset 恢复

4.然后你可以使用git commit 提交更改，建议一定要-m 附上更改信息！！！小心push的时候被管理员退回，当然，默认是推送到你自己的远程仓库，防止污染上游仓库。

5.PR一下，这一步指的是当你push到你自己的本地仓库，比如我就是push到本地仓库的，我不能直接push到上游，这时候我就需要PR，等待管理员审核，就可以merge到上游了。

省流：

使用 git add 将修改添加到暂存区。

使用 git commit 提交暂存区的修改到本地仓库。

使用 git push 将本地仓库的修改推送到远程仓库，与其他人共享。
